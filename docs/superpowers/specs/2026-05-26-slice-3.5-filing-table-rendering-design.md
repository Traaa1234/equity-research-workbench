# Slice 3.5 — Filing Table Rendering Polish Design

**Status:** Approved 2026-05-26. Implementation plan to follow.

**Builds on:** Slices 1, 2A (EDGAR ingestion + section parsing), 2B (LLM briefings), 2C (semantic search), 3 (RAG Q&A) — all shipped to production.

## Goal

Fix the messy text rendering on the filing reader page (`/stock/[ticker]/filings/[accession]`) where financial tables collapse into chaotic single-cell-per-line columns. After this slice, tables render as pipe-separated rows (`Net sales | $80,208 | $68,714`) in a monospace font, which preserves the relationships between cells and reads as a structured table even if not perfectly aligned visually.

This is opportunistic polish triggered by user feedback on real AAPL filing content. The briefing (Slice 2B), search (Slice 2C), and Q&A (Slice 3) features were unaffected by the messy text — only the raw section text view in the filing reader looked broken.

## Non-Goals

- **Proper markdown-table rendering** (with `| --- |` header separators + `react-markdown` swap). Bigger change for marginal aesthetic gain; defer if pipe-separated rows feel cramped.
- **Sanitized-HTML rendering** with `DOMPurify` / `rehype-raw`. Larger UI rewrite + new attack surface; defer.
- **Hybrid font rendering** (mono for tables, sans for prose). The monospace-everything trade-off is acceptable; revisit only on explicit feedback.
- **Auto-detecting financial tables into a structured `filing_tables` table**. Sophisticated parser work; separate product direction.
- **Python parser unit tests in CI (GitHub Actions)**. Carries forward from Slice 2A; remains deferred.
- **Parser-version column on `filings` / `filing_chunks`**. YAGNI until we change the parser a second time. Backfill remains a one-time manual operation.
- **Cascade FK from `chunk_embeddings` to `filing_chunks`**. Adds complexity; the explicit deletes in the backfill script are clearer about intent.
- **E2E Playwright tests**. Stack Auth ESM blocker carries from Slice 1C.

## Product

Today (Slice 3 era): the filing reader page shows section text like this:

```
March 28,
2026
March 29,
2025
Net sales:
  Products
$
80,208

$
68,714
```

After Slice 3.5, the same content renders as:

```
March 28, 2026 | March 29, 2025
Net sales:
Products | $80,208 | $68,714
Services | $26,645 | $25,236
```

Not a perfectly-aligned table, but immediately readable as rows with associated values. The monospace font ensures pipes line up vertically when columns have similar character widths.

**Trade-off accepted**: monospace makes tables readable but the surrounding prose (Risk Factors narrative, MD&A paragraphs) gets a more terminal-esque appearance. We accept this for v1 because the user-flagged issue was table chaos, not prose aesthetics.

## Parser changes

### `clean_html_to_text` + new `render_table_as_text` helper

Both `scripts/sec_fetch.py` (local) and `api/fallback/sec.py` (Vercel serverless) get the same change. They share the same `clean_html_to_text` function today.

```python
def render_table_as_text(table):
    """Render an HTML table as pipe-separated rows, one row per line.

    Empty cells are dropped to avoid noise from layout-padding <td>'s.
    Internal cell whitespace is collapsed to a single space.
    """
    rows = []
    for tr in table.find_all('tr'):
        cells = []
        for td in tr.find_all(['td', 'th']):
            cell_text = td.get_text(' ', strip=True)
            if cell_text:
                cells.append(cell_text)
        if cells:
            rows.append(' | '.join(cells))
    return '\n'.join(rows)


def clean_html_to_text(html: str) -> str:
    """Strip noise tags, render tables as pipe-separated rows, return plaintext."""
    soup = BeautifulSoup(html, 'html.parser')
    for tag in soup(['script', 'style', 'head', 'meta', 'link']):
        tag.decompose()

    # Drop Table of Contents tables (existing behavior — noise, not data)
    for table in soup.find_all('table'):
        text = table.get_text(' ', strip=True)
        if 'table of contents' in text.lower()[:200]:
            table.decompose()

    # NEW: replace remaining tables with their flat rendering before get_text()
    for table in soup.find_all('table'):
        rendered = render_table_as_text(table)
        replacement = soup.new_string('\n' + rendered + '\n')
        table.replace_with(replacement)

    text = soup.get_text('\n')
    text = re.sub(r'\r\n', '\n', text)
    text = re.sub(r'[ \t]+', ' ', text)
    text = re.sub(r'\n[ \t]+', '\n', text)
    text = re.sub(r'\n{3,}', '\n\n', text)
    return text.strip()
```

### Design choices baked in

| Choice | Reasoning |
|---|---|
| Pipe `\|` separator | Renders identifiably in any font; never appears meaningfully in financial filings. |
| Drop empty cells | Financial tables have padding `<td>&nbsp;</td>` for alignment; including these produces `\| \| \| \| $80,208`. Dropping makes rows readable. |
| `get_text(' ', strip=True)` per cell | Multi-line cells collapse into a single string so they can't break the row. |
| Process tables before global `get_text()` | The replacement is inserted as a `NavigableString`; outer `get_text('\n')` then composes naturally. |
| No header-row special-casing | Markdown-table separator (`--- \| ---`) was tempting; adds complexity for little gain. First row will read as a header regardless. |
| No nested-table handling | Rare in 10-K filings. If hit, the inner table falls back to the existing `get_text()` behavior. Acceptable. |

## UI rendering tweak

Single-line change in `app/(app)/stock/[ticker]/filings/[accession]/_components/section-nav.tsx`:

```tsx
// BEFORE
<pre className="whitespace-pre-wrap font-sans text-sm leading-relaxed">

// AFTER
<pre className="whitespace-pre-wrap font-mono text-xs leading-relaxed">
```

- `font-sans` → `font-mono` so pipes/columns align vertically
- `text-sm` → `text-xs` to fit wider rows without aggressive wrapping

Only the filing-reader section tabs are affected. The briefing card (Slice 2B), search snippets (Slice 2C), Ask answer (Slice 3) keep their existing fonts because they don't display table text.

## Backfill script — `scripts/reparse-existing.ts`

A new one-shot CLI: `pnpm reparse <TICKER>`. Mirrors the pattern of `scripts/embed-existing.ts` from Slice 2C. For each filing of the given ticker:

1. Re-fetch the HTML from the stored `primaryDocUrl` via `SecEdgarProviderImpl.fetchFiling()`
2. Run the new parser to produce fresh sections
3. In a single transaction: delete old `chunk_embeddings` rows + delete old `filing_chunks` rows + insert fresh `filing_chunks` rows
4. Outside the transaction (slow API call): call `EmbeddingsService.embedFiling()` to regenerate vectors from the fresh text
5. Log per-filing summary; per-filing try/catch so one failure doesn't abort the whole ticker

### Critical design choices

| Choice | Reasoning |
|---|---|
| **Re-fetch HTML from SEC instead of storing it locally** | We don't persist raw HTML, only parsed sections. The new parser must run against the original HTML, which only exists at SEC's URL. |
| **Transactional delete-then-insert for chunks** | Avoids any window where the filing has NO chunks (which would break the reader page mid-script). |
| **Delete `chunk_embeddings` explicitly even though there's no FK cascade to `filing_chunks`** | Old vectors reference chunks that are about to be deleted. If the new parser produces different `section_key` distributions, the old embeddings would orphan. Even if section_keys match, the underlying text changed → vector is stale. |
| **Re-embed OUTSIDE the transaction** | DashScope API calls take seconds; holding a DB transaction open that long is bad practice. `embedFiling()` is idempotent (ON CONFLICT DO NOTHING on PK), so it's safe to run post-commit. |
| **Per-filing try/catch + continue** | If filing N fails (e.g., SEC 503, DashScope rate limit), log and proceed to N+1. Best-effort, like the existing `embed-existing.ts`. |
| **Rate limiting via existing provider** | `SecEdgarProvider` already enforces SEC's 10/sec ceiling via `MIN_INTERVAL_SECONDS = 0.21` in the Python script. No additional script-level throttling needed. |

### Cost + runtime

| Per ticker | Value |
|---|---|
| SEC re-fetches | Free |
| DashScope re-embeddings | ~$0.025 (same as initial ingest) |
| Wall-clock | ~3-5 min (similar to Slice 2C's `embed-existing.ts`) |
| Re-runs needed? | No. One-shot per affected ticker. |

### Test infrastructure: Option B (inline assertion script)

No pytest setup. A standalone Python script that uses plain `assert` statements; exits non-zero if anything fails. Runnable via `py scripts/test_render_table.py` from CMD.

```python
# scripts/test_render_table.py
from bs4 import BeautifulSoup
from sec_fetch import render_table_as_text

def t(html):
    return render_table_as_text(BeautifulSoup(html, 'html.parser').find('table'))

def main():
    assert t('<table></table>') == ''
    assert t('<table><tr><td>A</td><td>B</td><td>C</td></tr></table>') == 'A | B | C'
    assert t('<table><tr><td>A</td><td>B</td></tr><tr><td>C</td><td>D</td></tr></table>') == 'A | B\nC | D'
    assert t('<table><tr><td></td><td>X</td><td>&nbsp;</td><td>Y</td></tr></table>') == 'X | Y'
    assert t('<table><tr><th>Q1</th><th>Q2</th></tr><tr><td>10</td><td>20</td></tr></table>') == 'Q1 | Q2\n10 | 20'
    assert t('<table><tr><td>  hello\n  world  </td></tr></table>') == 'hello world'
    print('All 6 tests passed.')

if __name__ == '__main__':
    main()
```

**No GitHub Actions wiring.** Run manually before commits. Slice 2.5 (form-type expansion) was originally going to add Python CI; that remains its responsibility.

## Implementation order

The plan that follows will execute in this order, each step committing independently:

1. **Update Python parsers** — `scripts/sec_fetch.py` + `api/fallback/sec.py` with new `render_table_as_text` helper + modified `clean_html_to_text`
2. **Inline Python unit tests** — `scripts/test_render_table.py` (6 assertions), runnable via `py scripts/test_render_table.py`
3. **UI font tweak** — `section-nav.tsx`: `font-sans text-sm` → `font-mono text-xs`
4. **Backfill script** — `scripts/reparse-existing.ts` + `package.json` entry
5. **Local backfill smoke** — run `pnpm reparse AAPL` against local DB, verify text changes + embeddings regenerate
6. **Push** — CI green, Vercel deploys the new Python serverless
7. **Production backfill** — run `pnpm reparse AAPL` from local pointed at production DB
8. **Browser smoke** — refresh `/stock/AAPL/filings/<accession>`, confirm readable tables

About 8 tasks across 4 milestones (M1: parser, M2: UI, M3: backfill, M4: deploy + verify). Smaller than past slices.
